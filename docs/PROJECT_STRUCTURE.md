# SHITBTI 项目结构与工程约束

## 1. 目标

本文件用于约束这个项目的工程组织方式，避免仓库被逐渐改造成一个不必要地复杂、难维护的前端工程。

核心原则：
- 轻量
- 清晰
- 低依赖
- 便于在 GitHub 网页端直接修改
- 适合 GitHub Pages 静态部署

---

## 2. 工程原则

### 2.1 总原则
这是一个静态 H5 项目，不是大型 Web 应用。
因此默认：
- 不上前端框架
- 不上构建系统
- 不上包管理器
- 不做前后端分离
- 不做复杂工程化抽象

### 2.2 允许的复杂度
允许：
- 基础 CSS 拆分
- 原生 JavaScript 模块拆分
- 数据文件拆分
- 文档目录
- 简单工具函数

不建议：
- React / Vue / Svelte
- TypeScript
- Webpack / Vite / Rollup
- Zustand / Redux
- Tailwind 作为构建依赖
- 任何需要安装依赖才能跑起来的方案

### 2.3 为什么这样规定
因为项目维护者当前更适合：
- 直接在 GitHub 网页编辑代码
- 随时在公司电脑打开项目地址查看效果
- 低门槛地调整题目、结果和文案

## 3. 推荐目录结构

```text
project-root/
├── index.html
├── 404.html
├── README.md
├── .gitignore
├── assets/
│   ├── css/
│   │   └── styles.css
│   ├── js/
│   │   ├── questions.js
│   │   ├── results.js
│   │   ├── app.js
│   │   └── utils.js
│   └── img/
│       ├── favicon.png
│       └── og-placeholder.png
└── docs/
    ├── PRD.md
    └── PROJECT_STRUCTURE.md
```

说明：
- `index.html`：页面入口，尽量保持干净。
- `assets/css/styles.css`：所有样式。
- `assets/js/questions.js`：题目数据。
- `assets/js/results.js`：结果数据。
- `assets/js/app.js`：页面逻辑、计算、交互。
- `assets/js/utils.js`：通用小工具，可选。
- `assets/img/`：图标、封面图等静态素材。
- `docs/`：产品和工程文档。

## 4. 文件职责要求

### 4.1 index.html
职责：
- 页面骨架
- 基础 meta 信息
- 挂载容器
- 引入 CSS 和 JS

要求：
- 不要塞太多业务逻辑
- 不要把所有内容继续写回一个超长单文件

### 4.2 styles.css
职责：
- 页面视觉样式
- 响应式布局
- 结果页与首页样式

要求：
- 保持命名统一
- 不追求过度 BEM 化
- 适度注释模块边界

### 4.3 questions.js
职责：
- 题目数组
- 每题选项与维度得分

要求：
- 纯数据导向
- 尽量不要混入 UI 逻辑
- 字段命名直观

### 4.4 results.js
职责：
- 人格结果映射
- 名称、总结、描述、标签、风险提示等

要求：
- 纯配置文件思路
- 易于文案修改

### 4.5 app.js
职责：
- 页面初始化
- 开始测试
- 渲染题目
- 处理答题
- 计算人格类型
- 渲染结果
- 复制结果文案

要求：
- 代码直观
- 函数拆分适度
- 不做过度抽象

### 4.6 utils.js
职责：
- 放轻量通用函数，例如：
  - DOM 查询封装
  - 复制文本
  - 简单模板拼接
  - 本地存储工具

要求：
- 可有可无
- 不为了“看起来专业”而硬拆文件

## 6. UI 与交互要求

### 6.1 首页
应当：
- 一眼知道这是测试产品
- 有明确开始按钮
- 有轻量研究机构包装
- 有简短免责声明

### 6.2 答题页
应当：
- 题目清晰
- 进度清晰
- 选项按钮足够大，适合手机点击
- 不要出现需要精细操作的小控件

### 6.3 结果页
应当：
- 第一屏就能看到结果名称
- 一句话总结要有记忆点
- 便于截图
- 按钮清楚：再测一次 / 复制结果

---

## 7. GitHub Pages 兼容要求

### 7.1 部署要求
项目必须可以：
- 直接作为静态站点部署
- 无需构建步骤
- 无需服务器
- 通过 GitHub Pages 正常访问

### 7.2 路径要求
- 尽量使用相对路径
- 避免必须依赖本地开发服务器才可运行的写法

### 7.3 文档要求
README 中必须写清楚：
- 本地如何直接查看
- GitHub Pages 如何开启
- 如何修改题目和结果

---

## 8. 内容修改便利性要求

项目未来高频修改的部分主要是：
- 题目文案
- 结果文案
- 标题与介绍
- 页面色彩和轻视觉细节

因此要求：
- 这些部分尽量集中管理
- 不要把文案散落在十几个函数里
- 不要把题目数据写进 HTML 模板字符串深处

---

## 9. 注释与命名要求

### 9.1 注释原则
- 写“为什么”比写“这是个函数”更有价值
- 关键模块可以有块级注释
- 不要满屏无意义注释

### 9.2 命名原则
- 尽量英文变量 + 中文内容
- 函数名直白清晰
- 不用炫技式命名

例如：
- `renderQuestion`
- `calculateResultCode`
- `copyResultText`
- `questionData`
- `resultMap`

---

## 10. 后续扩展预留

未来可能加入：
- 分享图生成
- 本地缓存答题结果
- 多版本题库
- 匿名统计
- 小程序迁移

要求当前工程：
- 为这些能力预留扩展空间
- 但不要提前为了这些假设需求而过度设计

原则：
**先把静态 H5 做对，再考虑升级。**
